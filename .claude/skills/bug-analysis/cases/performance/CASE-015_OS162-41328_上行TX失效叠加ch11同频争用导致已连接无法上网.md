# CASE-015: 上行 TX 发送失效叠加 2.4G ch11 同频争用导致"已连接但无法上网"（WiFi 未断开，TX_ERR 漫游全失败）

## 基本信息
- **案例ID**: CASE-015
- **分类**: performance
- **来源**: OS162-41328
- **创建时间**: 2026-06-16
- **匹配次数**: 0

## 现象描述
- 用户反馈 **WiFi 已连接但无法上网（connected but not working），重启手机后恢复正常**
- 机型 TECNO LK6，Android 16 / HiOS 16.2.0（LK6-16.2.0.145(OP001PF001AZ)FANS），MTK 平台
- 偶现；问题最后发生 2026-06-11 13:20:19（孟加拉国 Dhaka，UTC+6）
- 连接对象：SSID `SWIFT-TECH-5-FLOOR`，BSSID `bc:a9:93:54:f7:70`，2.4G ch11
- 关键：**WiFi 全程未断开**（13:16:35 连上后保持连接到日志结束），RSSI -57~-73dBm（中等，并不弱），但 PER 全程阵发性打满、吞吐≈0、DNS 解析失败

## 根因结论
**上行发送链路失效（TX path 异常）为主因 + 2.4G ch11 同频争用/干扰为加重因素，共同导致"已连接但无法上网"。WiFi 自身断连重连无法救回，最终需用户手动重启整机才恢复。**

机理链条：
1. RSSI 中等（-57~-70dBm）下，上行发送持续失败：`Tx fail = RTS fail + ACK fail`，其中 **ACK fail 占 fail 总数 82%**（发出 DATA 收不到 ACK）、RTS fail 持续累加（发 RTS 收不到 CTS）
2. 窗口增量量化：失败率 36%（>10% 异常）、重传率 43%（>40% 严重）、Rx total 几乎不增长（dup 卡住）、Rx rate 锁定最低档 65Mbps（速率坍塌）
3. 驱动判定发送异常 → 触发 `ROAM_DISCOVERY reason=TX_ERR` 漫游 **12 次（全部 TX_ERR，无一次因 RSSI 弱）**；但周围无更优 AP → `EVENT-ROAMING FAIL reason 1` **12 次 100% 失败**，反复全频扫描 churn
4. 扫描结果（scnFsmDumpScanDoneInfo）显示当前连接信道 **ch11 并不空闲**：24 次全频扫中 IdleTime 中位仅 885（<3000 繁忙）、MdrdyCnt 最高 213（>100 繁忙）、CU 反复达 22~40% Busy → ch11 同频争用/干扰明显，与 PER 阵发打满吻合
5. 业务层：`msys: DNS Resolution Failed` / `Failed to resolve using secondary DNS resolver` → 坐实"connected but not working"
6. 全程无 disconnect/deauth（WiFi 未断），重连后 PER 依旧高 → 故障持续至日志结束；"重启后正常"指用户手动重启整机（日志范围外，无直接佐证）

## 排查步骤
1. 框架层（main_log）：确认 WiFi 连接状态与业务面
   - `wpa_supplicant CTRL-EVENT-CONNECTED/DISCONNECTED` 确认是否真断开（本例 13:16:35 后未再断）
   - 搜 `DNS Resolution Failed` / `Failed to resolve` 坐实数据面不通
2. 驱动层（kernel_log `.localtime`，+6h 对齐 Dhaka 墙钟）查 `wlanLinkQualityMonitor`：
   - 拆 `Tx fail = RTS fail + ACK fail`，看 ACK fail 占比（本例 82%）、RTS fail 是否累加
   - 用连续两条采样算窗口增量：失败率、重传率（本例 36% / 43%）
   - 看 Rx total/rate 是否停滞坍塌、`PER(xx)` 是否阵发打满
3. 漫游分析：统计 `ROAM_DISCOVERY reason=` 分布（本例全 TX_ERR）与 `EVENT-ROAMING FAIL` 次数（本例 12 次全失败 reason 1）
4. **信道空闲度交叉验证**（关键，区别于"信道完全空闲"误判）：从 `scnFsmDumpScanDoneInfo` 提取当前连接信道的 IdleTime / MdrdyCnt / CU Value
   - 当前连接信道可由 kernel 单信道定向扫描（`Detected_Channel_Num = 1` + `Channel: <ch>`）的众数确定（本例 12 次单信道扫全为 ch11）
   - 对比 LinkQuality 的 `Congestion idle slot`（本机发包采样窗口）与 ScanDone 的 IdleTime（扫描 dwell 期间信道占用）：两者口径不同，不矛盾

## 建议措施
1. **排查上行 TX path 异常（首要）**：RSSI 中等下 ACK fail 占 82%、RTS fail 累加，需区分多少由空口争用导致、多少为 driver/fw TX path 卡死；提供该版本（LK6-16.2.0.145）fw TX 统计 / PA 状态定位 HW vs fw 逻辑
2. **评估 TX_ERR 漫游策略**：周围虽有 7 个 AP（`Total:7/22`）但 12 次漫游均 reason 1 失败；ch11 已繁忙时反复全频扫描 churn 进一步劣化吞吐 → 建议增加退避/自愈（触发 driver 内部 TX path reset）而非仅依赖整机重启
3. **AP/环境侧**：ch11 同频争用明显，建议路由器切换到更空闲信道（扫描显示 ch9 IdleTime=793、ch10=2611 略优）或改用 5G（ch149~161 IdleTime 4500+）
4. **自愈机制**：当上行连续失败 + 漫游连续失败达阈值时，driver 主动软复位 TX path，避免用户必须手动重启整机
