# CASE-008: 强信号下空口介质争用/同频干扰导致 Roblox 游戏加载极慢（WiFi 未断开）

## 基本信息
- **案例ID**: CASE-008
- **分类**: performance
- **来源**: LK7OS163-2102
- **创建时间**: 2026-06-08
- **匹配次数**: 0

## 现象描述
- 用户玩 Roblox 游戏时**加载极慢、一直转圈，有时完全无法加载**；用户主观反馈"网络连接良好"
- 机型 TECNO LK7，Android 16 / HiOS 16.3.0（LK7-16.3.0.120SP03），MTK 平台
- 应用 Roblox（com.roblox.client）v2.723.787
- 偶现；问题时间 2026-06-01 11:48:32（雅加达，UTC+7）
- 关键：WiFi 全程未断开、无漫游、无认证失败，RSSI -30dBm（优秀），**协商速率正常**

## 根因结论
**空口介质争用 / 同频干扰导致的信道拥塞——本机射频无缺陷（强信号、速率正常），但吞吐呈"突发-空闲"剧烈波动，导致低延迟/加载敏感业务（Roblox）卡顿。**

机理链条：
1. WiFi 连接稳定，RSSI -30dBm，协商速率正常（Tx 130~650 / Rx 650~722）→ 排除弱信号、rate-control 坍塌（与 CASE-006 的弱信号坍塌相反）
2. 吞吐量在数百 Mbps 与接近 0 之间剧烈跳变（突发-空闲模式）
3. driver 链路统计呈现典型拥塞特征：
   - `Tx fail = RTS fail + ACK fail` 恒成立，ACK fail（收不到 ACK）+ RTS fail（收不到 CTS）持续累加
   - **Rx error 高达 28~47%**（强信号下近半接收帧解不出来 → 空口被可听到却解不出的能量/帧占满）
   - PER 间歇突刺（均值 ~8.9，峰值 96），突刺时刻 idle slot diff 骤降（介质忙）、吞吐同步跌零
4. kernel 报大量 >80ms TX 延迟（退避/介质等待），ping 偶现 175ms（正常 5~24ms）
5. 全程无 disconnect/deauth，DNS 失败来自 Smart Caller 应用（非 Roblox，干扰项）

## 排查步骤
1. 框架层（main_log）确认 WiFi 全程连接、无断开/漫游/认证失败、RSSI -30dBm → 排除连接类与弱信号类问题
2. 驱动层（kernel_log `.localtime`，+7h 对齐雅加达墙钟）查 `wlanLinkQualityMonitor`：
   - 看 Tx rate / Rx rate 是否正常（本例正常，排除速率坍塌）
   - 拆 `Tx fail = RTS fail + ACK fail`，看 ACK fail / RTS fail 是否持续累加
   - **看 Rx error / Rx total 比例**（本例 28~47%，是比 ACK fail 更硬的拥塞/干扰指标）
   - 看 `PER(xx)` 是否间歇突刺、`Congestion(idle slot diff)` 是否忙闲突变
3. 时间线对齐：PER 突刺、idle diff 骤降、吞吐跌零聚集在问题时刻（雅加达 11:40~11:48 ≈ kernel 04:40~04:48）

## 建议措施
1. **空口抓包定位干扰源（首要）**：用第三方 sniffer 或 `connsys_picus_log`（PHY/空口）抓同信道占用，区分"邻居 AP 同频拥塞"还是"非 WiFi 干扰"，坐实 Rx error 28~47% 来源
2. AP/路由器侧：确认 "MAMOY" 信道使用率，切换空闲信道 / 改用 5GHz 频段
3. 多设备验证：检查同网络其他设备是否同样卡顿（拥塞应为多设备共性）
4. 排查 MTK 平台 DTIM 配置失败（code 4）的驱动/固件问题

## 数据局限
- 有 main_log + kernel_log + MDLog；本机 tcpdump/空口包未逐帧解析，干扰源类型（WiFi 同频 vs 非 WiFi）需空口抓包坐实
- 结论基于 driver 链路统计（PER/ACK fail/RTS fail/Rx error/Congestion），未从空口帧直接定位干扰发射源
