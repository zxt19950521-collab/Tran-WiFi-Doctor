# CASE-032: 微信视频号 2.4G 链路质量差 PER 突刺导致刷视频卡顿（WiFi 未断开）

## 基本信息
- **案例ID**: CASE-032
- **分类**: performance
- **来源**: CN6OS16-2382
- **创建时间**: 2026-07-06
- **匹配次数**: 0

## 现象描述
- 中国 VIP 粉丝反馈：更新版本后夜间 WiFi 经常无法刷新视频，切流量正常，同网旁路 iPhone 正常。
- 机型 TECNO CN6（XYZ），Android 16 / HiOS 16.3.0，`CN6-16.3.0.132SP01(OP002PF001AZ)FANS`，MTK 平台。
- 应用：微信 `com.tencent.mm` 视频号 `FinderShareFeedRelUI`，版本 `8.0.69`。
- 用户体感问题锚点 **22:29** 刷视频卡顿无法播放；录屏 `Screen_Recording_20260704_222929.mp4`、TagLog `223027` 对齐。
- 关键：WiFi 全程保持连接并多次 `Wifi network validated`，不是断连、DHCP、认证失败；切流量后正常。

## 根因结论
**2.4G ch8 空口链路质量间歇恶化（PER 突刺 78~100%、PHY 速率坍塌至 65/10~55 Mbps、ACK fail 主导的上行可靠性差），叠加邻道 ch9 CU 偏高（54~97，约 21%~38% busy），导致系统 Ping/Probe 大量 1s 超时与 DNS/HTTP 失败；微信视频号拉流中断，表现为「WiFi 连着但刷不了视频」。ch8 同信道 AP 不密集（BAndPCnt 1~3），非典型同频 AP 扎堆拥塞。**

机理链条：
1. 用户连接 `CMCC-bbvd`（2.4G ch8，BSSID `50:46:4a:58:e6:fc`），环境存在配对 5G 热点 `CMCC-bbvd-5G` 但本机始终未切换。
2. `22:18~22:19` 关 WiFi 前 Ping 失败率约 **75%**（27 失败 / 9 成功），WiFi 仍 validated；`22:19:05~11` kernel `PER(84~100)`、速率 **65/65 Mbps**、RSSI **-68~-69 dBm**。
3. 用户 `22:19:12` 手动关 WiFi 切流量；`22:28:09` 重开 WiFi 后 `22:29` 刷视频仍失败。
4. `22:29:00~13` kernel `PER(78~92)`、速率 **65/10~55 Mbps**、ACK fail 占 fail 主体；`22:29:03` Ping 1000ms 超时；`22:29:04` 微信 `AudioTrack` stopped/released，视频播放中断。
5. 全信道扫描：ch8 `BAndPCnt` 通常 **1~3**（AP 不密集），但邻道 **ch9 CU 54~97**；ch8 `MdrdyCnt` 有突发（52~121），存在短时空口活动。
6. Rx error 低、Congestion idle slot 非极小 → 更像「链路质量差/上行 ACK 失败」而非典型信道极度拥塞；与 CASE-020 模式一致，区别是本例 22:29 RSSI 约 **-57 dBm** 中等偏好、用户明确在刷微信视频号。

## 排查步骤
1. 框架层确认是否真断网：
   - 搜 `Wifi network validated`、`CTRL-EVENT-DISCONNECTED`、`Wifi network lost`、`CMD_IP_CONFIGURATION_LOST`。
   - 区分用户手动 `setWifiEnabled`（本例 `22:19:12` 关、`22:28:09` 开）与意外断连。
2. 业务时间锚点对齐：
   - Jira 填写 `22:28:09` 为最后问题时间（重开 WiFi）。
   - 录屏 `222929` 与前台 `FinderShareFeedRelUI` 对齐 **22:29** 为刷视频卡顿主锚点。
3. 网络探测：
   - 查 `WifiStatistics:WifiNetworkCheck` 的 `Ping result` / `Probe result`。
   - 本例 validated 与 1000ms 超时并存；`22:29:03/13/23/24/43` 连续失败。
4. 应用层对齐：
   - 搜 `FinderShareFeedRelUI`、`AudioTrack`、`com.tencent.mm`。
   - 本例 `22:29:04` 音视频播放停止与 Ping 超时同秒。
5. 驱动链路质量：
   - kernel `.localtime` 查 `wlanLinkQualityMonitor` 的 Tx/Rx/PER/ACK fail/RTS fail。
   - 查 `mtk_cfg80211_get_station` 的 RSSI、link speed。
6. 信道环境：
   - 从 `scnFsmDumpScanDoneInfo` 提取 ch8 的 `BAndPCnt`、`IdleTime`、`MdrdyCnt`、`CU Value`。
   - 同时看邻道 ch9 CU；确认是否存在 `SSID-5G` 而未连接。
7. 排除 VPN/应用单独故障：
   - 无 VPN 日志；切流量恢复 → 排除纯应用 bug。

## 建议措施
1. 用户侧：优先改连 **`CMCC-bbvd-5G`**；靠近路由器；问题出现时勿立即关 WiFi，直接抓 TagLog + NTLog。
2. 测试侧：同位置对比 **CN6 vs iPhone**（确认 iPhone 是否走 5G）；路由器 2.4G 改信道避开 ch9 高 CU 时段；补空口 sniffer。
3. 研发侧：参考 CASE-020 向 MTK 提 case，附 `22:19:05`/`22:29:00` PER 突刺与 ACK fail 统计；查 RA 是否在 RSSI -55~-70 时过度降速。
4. 框架侧：高 Ping 失败率时评估网络评分/用户提示，避免「显示 validated 实际不可用」。
5. 回复口径：WiFi 未断开且多次 validated，但 22:29 驱动 PER 高、Ping 超时、视频号播放中断，与弱链路高时延一致；不建议按微信单独应用问题关闭。

## 数据局限
- 无空口 sniffer，无法逐帧确认 AP 是否收到数据帧/是否回 ACK。
- tcpdump（NTLog）未深入解析。

- 版本更新与问题的因果关系需更多样本；当晚 `22:02` 起已有零星 Ping 超时。
