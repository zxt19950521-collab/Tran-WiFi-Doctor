# CASE-013: 双频mesh下5G漫游被Auth Status33(带宽不足)拒绝，困在上行恶化的2.4G导致游戏卡顿（WiFi未断开）

## 现象描述
- 印尼粉丝反馈（原声）："tiba tiba wifi nya nge lag sendiri tanpa ada masalah dari wifi karna aku coba di hp lain, gada masalah dengan wifi" —— **WiFi 突然自己卡顿；换别的手机连同一 WiFi 都正常**
- 机型 LK7（TECNO），MTK mt6878，hios16.3.0 / Android 16，`LK7-16.3.0.120SP03(OP003PF001AZ)FANS`
- 偶现；工单登记问题时间 2026-06-09 18:51:03（雅加达 UTC+7），**用户录屏确认卡顿最严重在 ~18:54**（游戏卡死谷底）
- 连接双频 mesh "UCOK"：5G `84:07:fa:22:33:2c`(ch161) / 2.4G `84:07:fa:22:33:28`
- 关键：WiFi 全程未断开（0 次 disconnection），属"卡顿而非断网"

## 根因结论
**设备在双频 mesh 下被"困"在一条上行严重恶化的 2.4G 链路上，由两个独立因素叠加：**

1. **5G 漫游持续被拒（核心，AP 侧）**：漫游算法 87 次判定 5G（...2c, RSSI -58~-64）为更优并发起漫游，但全部被 5G AP 在**认证阶段**以 `Status Code = 33` 拒绝（18:46–18:56 覆盖问题点，18:54 单分钟 11 次）。
   - Status 33 = `STATUS_CODE_ASSOC_DENIED_BANDWIDTH`（MTK 驱动与 IEEE 一致）：*Assoc denied due to insufficient bandwidth to handle another QSTA* —— **AP 带宽/资源不足，无法再接纳一个 STA**，属 AP 侧接纳控制/负载拒绝（或 AP 借此码做 band-steering）。
   - 结果：`Can't roam out, try blocklist`，无法迁移 5G，被迫滞留 2.4G。
2. **被困的 2.4G 上行链路失效**：信道空闲（idle slot 24.6 亿，非拥塞）、RSSI 中等（-40~-65，非弱信号），仍出现大量 **RTS fail(21043) + ACK fail(16285)**，上行帧到达 AP 失败；Tx 速率被压到 65Mbps、Rx 塌到 10-55、PER 阵发飙至 100% → 吞吐间歇塌陷 → 游戏卡顿（18:54 谷底）。

机理链条：2.4G 上行恶化 + 5G 被 Status33 拒 → 想走走不掉，反复漫游 churn（14min 内 DISCOVERY 58 / TRANSITION 88）→ 吞吐间歇塌陷 → 18:54 游戏卡死；全程 BSSID 稳定、无 disconnect/deauth。

## 排查步骤
1. main_log wpa_supplicant 还原连接：UCOK 双频，问题窗在 2.4G ...28，反复重关联。
2. kernel `wlanLinkQualityMonitor` 提取链路统计：RTS fail/ACK fail 高、PER 阵发 100%、idle slot 24 亿（信道空闲，非拥塞）→ 场景 D 上行链路失效型。
3. kernel `mtk_cfg80211_get_station`：RSSI 中等(-40~-65)、Tx 速率被压 65、连接 BSSID 为 2.4G ...28。
4. kernel `saaFsmRunEventRxAuth` 发现 **Auth Status 33 ×87**，全来自 5G ...2c，且 `Can't roam out` → 定位 5G 漫游被 AP 以"带宽不足"拒绝。
5. 时间线对齐：18:54 为 roaming churn 最密集(53次/2min) + Status33 最频繁(11次/min) + Rx 速率塌到 10Mbps + PER 100% 的谷底，与用户录屏一致。

## Status Code 参考
```c
/* Assoc denied due to insufficient bandwidth to handle another QSTA */
#define STATUS_CODE_ASSOC_DENIED_BANDWIDTH          33
```
IEEE 802.11 标准与 MTK 驱动定义一致：33 = AP 带宽/资源不足，无法再接纳一个（QoS）STA。

## 建议措施
1. **抓 5G 漫游被拒根因（首要）**：复现时同抓 sniffer（空口）+ AP 侧日志/配置，确认 5G AP 返回 Status 33 是真满载（连接数/负载达上限）还是策略性 band-steering 赶客。
2. **核对本机型漫游策略**：88 次/14min 漫游过频，且明知 5G 被拒仍反复 churn；评估被拒后是否应稳定保持 2.4G 并退避，减少全频扫描对游戏数据的中断。
3. **排查 2.4G 上行失效**：RTS/ACK fail 高而信道空闲、信号中等 → 查本机 2.4G 上行 RF/天线/功率（对比参考机同点位 RTS/ACK fail），及是否存在 hidden node / 本地干扰。
4. **与对比机交叉验证**：用户称"别的手机正常"，建议对比机在同 AP 同位置抓 log，对比：①是否成功连 5G（未被 33 拒）；②2.4G 上行 RTS/ACK fail 是否同样高 → 区分"本机型问题"还是"AP 环境/策略"。
5. **不建议简单归类为应用问题**：日志证据指向 WiFi 链路层/漫游层（上行失败 + 5G 被拒），应用卡顿是 WiFi 吞吐塌陷的结果。
